## 二、测试框架与自动化能力

### Q4: 你有没有从 0 到 1 搭建过测试框架?

**回答要点:**

- 有。曾参与搭建面向分布式系统的自动化测试和诊断框架。
- 框架分层包括:核心能力库、远程执行引擎、自动化用例层、CI 流水线、结果汇总和报告层。
- 核心能力包括:集群连接、SSH 命令执行、远程脚本执行、测试数据采集、指标监控、结果上传、故障注入封装。
- 框架目标是让测试人员用统一 API 编写用例,屏蔽底层环境差异。

### Q5: 你搭测试框架时最先考虑什么?

**回答要点:**

- 先明确被测系统特点:是否分布式、是否多节点、是否需要远程执行、是否需要和 CI/测试管理平台集成。
- 再设计目录结构和分层:公共库、测试用例、配置管理、执行入口、报告模块、CI 配置。
- 然后逐步建设核心能力:连接管理、环境初始化、用例执行、日志采集、结果汇总。

### Q6: 你的测试框架如何管理配置?

**回答要点:**

- 把环境信息、账号、集群地址、运行模式、测试参数集中管理。
- 支持默认配置、命令行覆盖、运行时注入。
- 避免在用例里硬编码 IP、账号、路径和环境信息。

### Q7: 如何避免测试代码重复?

**回答要点:**

- 把通用操作封装到公共库,例如集群连接、服务重启、日志查询、指标采集、数据准备、校验逻辑。
- Robot Framework 中封装关键字,Pytest 中封装 fixture/helper 函数。
- 对重复的测试流程进行模板化,只把差异参数暴露给用例。

### Q8: 你的自动化用例怎么接入 CI?

**回答要点:**

- 通过 Jenkins Job 定时或按代码提交触发。
- 按 Tier1/Tier2 或 smoke/regression 分层执行。
- 执行后归档日志、测试报告、JUnit XML、指标文件。
- 失败时保留现场信息,方便定位。

### Q9: 如何提高自动化测试稳定性?

**回答要点:**

- 减少硬编码等待,使用 polling / retry / timeout 机制。
- 测试前做环境健康检查,测试后做清理。
- 对异步后台任务使用状态轮询,不要直接 sleep。
- 对 flaky case 分类处理:环境问题、产品问题、脚本问题分别定位。

## 三、Pytest / Robot Framework

### Q10: Pytest 和 Robot Framework 你怎么选择?

**回答要点:**

- Pytest 更适合 Python 代码级测试、复杂逻辑封装、参数化、fixture、CI 集成。
- Robot Framework 更适合关键字驱动、可读性强、业务流程型测试,非开发人员也容易看懂。
- 如果项目需要复杂编程能力和灵活断言,我倾向 Pytest;如果更关注业务流程表达和可读性,Robot 更合适。

### Q11: Pytest 常用特性有哪些?

**回答要点:** fixture、parametrize、marker、conftest.py、hook、JUnit XML、pytest-html/report。

### Q12: Robot Framework 的优势是什么?

**回答要点:** 关键字驱动、可读性强、报告友好、业务流程表达清晰、可通过 Python 扩展关键字。

## 四、CI/CD 与质量保障

### Q13: 你维护过 Jenkins 流水线吗?

**回答要点:**

- 维护过持续测试流水线,包括自动化 Job 接入、参数配置、报告归档、失败通知和结果分析。
- 按测试层级拆分执行:快速冒烟、功能回归、性能测试、升级测试、故障注入测试。
- 通过流水线提升测试执行频率和回归效率。

### Q14: 如何设计 Tier1/Tier2 测试?

**回答要点:**

- Tier1:运行时间短、覆盖核心功能、每次提交或每天频繁跑。
- Tier2:覆盖更复杂场景,包括性能、升级、故障注入、长稳测试,运行时间更长。
- 目标是在反馈速度和覆盖深度之间做平衡。

### Q15: 如何处理 CI 失败?

**回答要点:**

- 先判断失败类型:代码缺陷、环境问题、测试脚本问题、依赖服务问题。
- 查看失败日志、测试报告、历史趋势和最近代码变更。
- 如果是 flaky,要收集证据并分类,不能简单忽略。
- 对真实缺陷及时创建问题并跟进修复。

## 五、分布式系统测试

### Q16: 分布式系统测试和普通应用测试有什么不同?

**回答要点:** 多节点协同、数据一致性、故障恢复、升级兼容、性能退化、长期稳定性,需要结合日志、指标、API 和内部状态综合判断。

### Q17: 你怎么设计故障注入测试?

**回答要点:** 分析架构和故障点;设计进程 kill、节点重启、网络异常、磁盘异常、服务不可用、升级中断;注入前记录 baseline,注入后观察降级表现,恢复后验证数据一致性。

### Q18: 如何验证数据一致性?

**回答要点:** checksum/元数据校验、故障前中后读写验证、后台恢复任务状态跟踪、API/日志/内部状态交叉验证。

### Q19: 升级测试关注什么?

**回答要点:** 升级前健康检查、升级中可用性、升级后功能兼容/数据一致性/配置兼容/性能回归、滚动升级新旧版本混跑。

## 七、AI / RAG / LLM 测试

### Q23: 你做过哪些 AI 相关项目?

**回答要点:** AI Agent、测试用例生成、故障诊断、CI 失败分析、PR 辅助处理、RAG 知识库、本地 RAG 评测框架。

### Q24: 什么是 RAG?

**回答要点:** Retrieval-Augmented Generation,先从知识库检索相关上下文,再让 LLM 基于上下文回答,减少凭空回答并提升可追踪性。

### Q25: 如何评估 RAG 系统质量?

**回答要点:** 构建 golden dataset,评估 retrieval 是否找对 context,评估 answer relevancy、correctness、faithfulness、hallucination,同时看分数和 judge reason。

### Q26: 什么是 hallucination?怎么检测?

**回答要点:** 模型生成了上下文不支持或事实错误的内容。在 RAG 中把答案和 retrieved context 对比,如果答案包含 context 没有的事实,就是不忠实或幻觉。

### Q27: DeepEval 是什么?

**回答要点:** DeepEval 是 LLM 应用评测框架,类似 LLM 应用的单元测试工具,可用 Answer Relevancy、Faithfulness、Hallucination、GEval 等指标评估模型输出。

### Q28: LLM-as-a-Judge 有什么风险?

**回答要点:** Judge model 可能不稳定或误判;需要清晰 criteria、保留 judge reason、人工复核,最好 agent model 和 judge model 分开。

### Q29: 你如何构建 golden dataset?

**回答要点:** 从知识库和真实业务问题出发,覆盖核心概念、高频问题、边界情况和负向问题;expected answer 必须有文档依据;失败后区分 dataset/retrieval/generation/judge 问题。
