# 测试框架搭建 面试题总结

这份文档整理"从 0 到 1 搭建测试框架"这个话题下,面试官可能会问的问题,以及参考答案要点。配合 `how-to-build-a-test-framework.md` 一起看效果更好。

答案里所有内容都做了通用化处理,不涉及任何公司或产品信息。

---

## 一、总体设计类

### Q1: 这个测试框架是你自己搭的吗?讲讲整体设计。

**要点:**

- 是从 0 到 1 自己设计的,不是套模板。
- 核心定位:面向运行在 Linux 集群上的被测系统,既是诊断工具箱,又是自动化测试套件。
- 分层:`lib`(纯函数能力库)、`remote`(远程脚本引擎)、`automation`(测试用例)、`ci`(流水线 + 脚本生成)、`dist`(生成物)、`test`(框架自身单测)。
- 设计哲学是 **Function First**:用纯函数 + 装饰器组合能力,而不是庞大继承体系。
- 四大核心机制:Context 上下文、Mode 多模式分发、跨平台降级、远程 Python 代码生成。

### Q2: 你为什么不用现成的框架(pytest/robot framework)从头造轮子?

**要点:**

- 其实**没有完全造轮子**,测试执行层仍然基于 pytest/unittest。
- 自研的是**领域能力层**:远程连接、跨站点脚本执行、把普通函数打包成零依赖脚本——这些是现成框架不提供的。
- 现成框架解决"怎么组织和运行用例",自研部分解决"怎么在受限的远程环境里拿到诊断数据"。
- 是"在成熟框架之上做领域封装",而不是重复造已有的轮子。这也是一个成熟工程师该有的判断。

### Q3: 框架最难的部分是什么?

**要点:**

- 远程 Python 代码生成:用 AST 静态分析一个函数的所有依赖(变量、子函数、import),再拼成一个自包含单文件脚本。
- 难点在于:要正确处理作用域、递归依赖、跨 Python 版本 AST 差异,还要拦截不允许在远程使用的第三方库。

## 二、Context / 配置管理类

### Q4: 参数透传问题你是怎么解决的?

**要点:**

- 引入 Context 机制:把"在固定上下文里不变的值"声明成函数,用 `@ctx` 装饰成懒加载单例。
- 业务函数直接调用 context 函数取值,不用一层层传参。
- 支持运行时 `update` 注入、`has` 判断、无值时尽早抛错。

### Q5: 配置项默认值、运行时注入、命令行覆盖,怎么统一?

**要点:**

- 在 `@ctx` 之上再叠一个 `@ctx_arg`。
- 框架有一个全局参数注册表,每个 `@ctx_arg` 注册"如何 add_argument"和"解析后如何写回 context"。
- 结果:配置项只声明一次,就同时拥有默认值、运行时注入、命令行覆盖三种能力。

### Q6: Context 是全局单例,会不会有并发/测试污染问题?

**要点:**

- 会。缓解方式:提供 `update` 显式重置、`has` 判断状态;测试里在 `setUp` 统一初始化。
- 需要并发隔离的资源(如锁)本身也用 context 持有,并配合线程锁使用。
- 这是一个可以坦诚讨论的权衡:换来了极简的调用体验,代价是需要规范初始化时机。

## 三、多模式 / 架构模式类

### Q7: 同一个操作在不同环境有不同实现,你怎么设计?

**要点:**

- 用 `@current_mode` + `register_mode` 做多重分发:一个函数名,多套实现,运行时按 `mode` context 选择。
- 本质是"策略模式 + 多重分发"。
- 调用方永远只写 `list_nodes()`,不写一堆 `if mode == ...`。
- 新增模式只需补一个 `register_mode`,符合开闭原则。

### Q8: 用了哪些设计模式?

**要点:**

- **策略模式 + 多重分发**: Mode 机制。
- **装饰器模式**: context、结果上报、工具函数标记都用装饰器叠加横切能力。
- **懒加载单例**: context 值第一次访问才计算并缓存。
- **模板方法**: `TestCase.setUp` 统一连接编排。
- **空对象模式**: 跨平台降级时用占位模块代替不可用模块。

## 四、远程执行 / 代码生成类

### Q9: 被测节点上没有你的代码、没有第三方库,你怎么在上面跑逻辑?

**要点:**

- 用远程 Python:给函数打 `@tool_function` 标记,框架把它生成成一个**自包含单文件脚本**。
- 生成时用 AST 递归收集依赖:标准库/白名单 → 允许 import;第三方非白名单 → 报错;项目内函数 → 递归打包;普通变量 → 序列化进脚本。
- 有参数就自动生成 argparse。
- 最终直接 `python3 xxx.tool.py` 就能跑,不依赖 pip。

### Q10: 远程执行的结果怎么拿回来?

**要点:**

- 用 `@tool_meta(saveOutput="result.json")` 让脚本把返回值写成 JSON。
- 封装 `ssh_remote_node(ip).exec(fn, **kwargs)`:上传脚本 → SSH 执行 → 下载 JSON → 反序列化返回。
- 调用方写起来像调本地函数,底层细节全隐藏。

### Q11: 怎么保证生成的脚本能在旧版本 Python 上运行?

**要点:**

- 生成流程里有专门的 Clean up 阶段,处理不同 Python 版本的 AST 节点差异。
- 维护白名单模块,禁用远程环境不存在的第三方库(遇到就直接报错,而不是运行时才崩)。

### Q12: 怎么防止工具函数改了但生成脚本忘记更新?

**要点:**

- CI 里加一道"生成物一致性闸门":重新生成 `dist/` 后跑 `git diff --exit-code`。
- 不一致就失败,并提示去跑刷新脚本的 Job。
- 把"生成物必须和源码同步"变成自动化红线。

## 五、跨平台 / 工程细节类

### Q13: 开发机是 Windows,被测是 Linux,依赖冲突怎么办?

**要点:**

- 平台专用模块(如 `fcntl`)用 try/except import 做优雅降级:导不进来就替换成占位模块。
- 代码在 Windows 上照样能 import、跑单测,只有真正调用该能力时才失败。
- 避免"一个平台依赖拖垮整个框架在开发机上的可用性"。

### Q14: 频繁上传同一个脚本很慢,怎么优化?

**要点:**

- 上传前先算本地文件 sha256,和远程 `.sha256` 缓存比对,一样就跳过上传。
- 配合线程锁避免并发重复上传。
- 小优化,但在反复调试场景下提速明显。

### Q15: 为什么强制所有函数名全局唯一?

**要点:**

- 因为脚本生成要靠函数名定位和拼接代码,重名会歧义。
- 在 CI 里扫描所有模块,发现重名直接报错,把约束前置到提交阶段。

## 六、测试执行 / 结果管理类

### Q16: 用例结果怎么同步到测试管理平台?

**要点:**

- 用 `@qtest_case` 装饰器把"执行用例"和"上报结果"解耦。
- 执行前按 `模块/特性/类型` 层级**幂等**创建(存在复用,不存在才建)Test Case/Cycle/Suite/Run。
- 执行后按 pass/fail 写回执行记录,同时存进全局结果列表。
- 用例代码本身保持干净,不掺杂上报逻辑。

### Q17: 参数化用例怎么在平台上体现成多条记录?

**要点:**

- 用例名支持传函数,根据参数动态生成名字。
- 一个 `test_xxx` 里循环跑多组数据,每组在平台上都是独立记录,失败能精确定位到哪组参数。

### Q18: pytest 有哪些关键配置?

**要点:**

- `python_functions` 设成一个永不匹配的名字,关掉对普通 `test_` 函数的自动收集,只跑 TestCase 方法,避免误收集。
- `--capture=tee-sys` 实时输出 stdout,长任务能看进度。
- 用 `markers` 给用例分层分类(冒烟/回归/按硬件),运行时用 `-m` 选择性执行。

### Q19: 整个测试会话结束后怎么汇总结果?

**要点:**

- 用 `conftest.py` 的 `pytest_sessionfinish` 钩子。
- 把全局结果列表汇总成 JSON(总数/通过数/明细),供流水线归档。

## 七、CI/CD 类

### Q20: 你的 CI 流水线有哪些阶段?

**要点:**

1. CodeStyle:代码风格检查,不合格失败并回帖 PR。
2. Test:先生成脚本,再跑单测并产出 JUnit XML。
3. Check:生成物一致性闸门(git diff)。
- 全部在统一 devkit Docker 镜像里跑,保证本地/CI/被测三方环境一致。

### Q21: 怎么保证"在我机器上是好的"这类问题不发生?

**要点:**

- 统一 Docker devkit 镜像:锁定 Python 版本和依赖。
- 跨平台降级保证 Windows/Linux 都能 import。
- CI 强制风格 + 单测 + 生成物一致性三道闸门。

---

## 八、反思 / 权衡类(高级问题)

### Q22: 这个框架有什么缺点或你会怎么改进?

**要点(诚实回答更加分):**

- 全局单例 context 在并发和测试隔离上需要规范,理想可以引入作用域化的上下文。
- 远程代码生成对"动态特性"(反射、闭包捕获复杂对象)支持有限,依赖必须是可静态分析的。
- 白名单模块机制维护成本高,新依赖要手动登记。
- `dist/` 生成物入库会让 diff 变大,但换来一致性校验,是有意识的取舍。

### Q23: 如果让你重做,你会保留什么、改什么?

**要点:**

- 保留:Function First、Context、远程代码生成这三个核心思想——它们解决的是真实痛点。
- 改进:context 作用域化;远程脚本生成加更完善的依赖分析和报错提示;把测试平台上报做成可插拔(适配不同平台)。

### Q24: 这个框架体现了你哪些能力?

**要点:**

- 抽象能力:把重复痛点抽象成 context / mode / 远程执行三个通用机制。
- 工程能力:AST 元编程、跨平台处理、CI 闸门设计。
- 落地能力:框架不是玩具,支撑了真实的自动化测试用例和结果上报闭环。
- 判断力:该复用 pytest 就复用,该自研的领域能力才自研。
